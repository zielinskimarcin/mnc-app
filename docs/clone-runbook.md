# Clone Runbook

Date: 2026-05-11

This template is intended to be cloned as one isolated product per restaurant during the first commercial phase.
Each client should get its own mobile app build, admin dashboard deployment, and Supabase project.

## Recommended Deployment Model

Use one Supabase project per restaurant and one admin deployment per restaurant.

For the first 10-20 clients, avoid one shared multi-tenant admin dashboard at `/mnc`, `/pogodna`, and similar paths. Separate deployments are simpler to reason about, safer for owners, and reduce the chance of one client seeing another client's data or configuration. A shared multi-tenant dashboard can come later when tenant isolation, billing, support, and migrations are mature.

Good early model:

- `client-app` repo/folder copied from `concept-app`
- `client-admin` repo/folder copied from `mnc-admin`
- one client Supabase project
- one client EAS project/app identifier
- one client admin URL or GitHub Pages/Vercel deployment

## Client Config Files

Mobile app:

- `src/config/tenant.ts`: brand name, app scheme used by OAuth redirects, menu categories, tab labels/icons, loyalty copy, reward threshold
- `src/ui/theme.ts` and screen/component styles: visual system, colors, spacing, typography, borders
- `app.json`: native Expo/App Store identity: app name, slug, scheme, bundle identifier, icons, EAS project id
- `assets/`: client icon, splash, screenshots
- `.env` / EAS env: `EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_ANON_KEY`

Admin:

- `src/tenant.ts`: dashboard title, GitHub Pages/Vite base path, menu categories, default push screen, tab labels
- `VITE_ADMIN_BASE_PATH`: optional deploy override for Vite base path
- `.env`: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`

Menu category keys in mobile/admin config must match `menu_items.category` values in Supabase.

## Supabase Setup

For a fresh restaurant:

1. Create a new Supabase project.
2. Apply all migrations from `supabase/migrations`.
3. Deploy Edge Functions from `supabase/functions`.
4. Set required secrets for push sending and cron execution.
5. Configure `pg_cron` scheduled push runner, or keep the GitHub Action as a manual fallback only.
6. Create the first admin user and set their `profiles.role` to `admin`.
7. Confirm public writes are closed for `app_config`, `push_tokens`, `push_opens`, and `loyalty_events`.
8. Confirm `loyalty_events` RLS is enabled and staff/admin can read the point adjustment history.

## Auth Provider Checklist

Email/password works as the baseline login path.

Google:

- Create the correct Google OAuth client(s) for the app.
- Add the Expo/AuthSession redirect URI generated by the app scheme to Supabase Auth redirect allowlist.
- Keep `src/config/tenant.ts` `appScheme` aligned with `app.json` `expo.scheme`.
- Verify on a physical TestFlight/development build, not only Expo Go.

Apple:

- Enable Sign in with Apple for the iOS bundle identifier in Apple Developer.
- Configure Supabase Apple provider so the accepted audience/client identifier matches the native iOS app.
- For Laura/MNC, the failing live error on 2026-05-11 was `Unacceptable audience in id_token: [com.greenvoi.conceptapp]`. That means the Apple/Supabase provider configuration must be fixed before Apple login can be called verified.
- Verify on a physical iPhone TestFlight/development build.

Do not mark Google/Apple as production-ready for a clone until the native build login flow creates a Supabase session and a row appears in `auth.identities` for that provider.

## Smoke Test Per Clone

Mobile:

- app boots with no blank screen
- email sign up/login
- Google login if enabled
- Apple login if enabled
- push permission request
- push token saved through `register_push_token`
- menu loads and pull-to-refresh works
- profile loads and name update works
- points screen loads and code modal opens
- delete account flow works on a test user

Admin:

- dashboard refuses missing env with config error screen
- admin login works
- non-staff user is denied access
- menu add/edit/delete works
- point add/remove works
- point operation appears in `loyalty_events` and the dashboard history
- immediate push sends
- scheduled push creates a job
- scheduled runner sends due jobs
- push history and open counts load

## Current Product Boundaries

The app is suitable for controlled restaurant pilots.

Before claiming mature analytics, add deeper owner-facing metrics: active users, repeat customers, reward redemptions, push conversion, menu engagement, and retention cohorts.

Before scaling to 50-100 clients, add a scripted clone command or checklist automation for Supabase migrations, Edge Function deployment, EAS env, admin deploy env, and first-admin creation.
